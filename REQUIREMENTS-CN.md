# 需求文档

## 背景

NavSphere 已具备导航数据管理、站点管理、图片资源管理、历史版本、图标缓存、站点信息自动获取和系统状态检查能力。深度使用后台后，当前主要问题不是能力缺失，而是部分维护流程仍偏工程化：用户能完成操作，但过程反馈、风险控制和批量维护记录还不够直观。

本文档记录后续面向管理员体验的改进需求。

## 用户痛点

1. 数据管理页职责过重。
   - 当前页面同时包含 JSON 编辑、保存、下载、上传和恢复历史。普通管理员容易把它理解成“危险控制台”。

2. JSON 编辑入口风险感强。
   - 直接编辑完整导航 JSON 适合高级维护，但不适合日常运营。即使有校验和历史版本，也仍然容易误操作。

3. 站点列表批量操作结果不可追溯。
   - 管理大量站点时，用户常需要批量启用/禁用、移动分类、设置公开/私密等操作。任务执行完后，目前只能看到即时反馈，无法长期追溯本次具体修改了哪些站点。

4. 资源库和自动缓存概念容易混淆。
   - 用户不关心 Blob、KV asset、favicon cache key 等实现细节，更希望统一理解为“图片资源”和“自动缓存图标”。

5. 系统配置状态不够直观。
   - Redis/KV、Vercel Blob 等配置缺失时，用户需要从错误信息判断能力是否可用。后台应直接展示当前存储能力和缺失项。

## 需求列表

### P1：数据管理页拆分为更清晰的维护区域（已完成基础能力）

降低数据管理页的认知负担，把高风险能力和日常维护能力区分开。

当前状态：基础拆分已完成，数据管理页按“日常维护 / 备份恢复 / 高级 JSON”分区；高级 JSON 作为独立高级入口，保存前展示数据变化摘要，高风险变更需要再次确认。

功能要求：

- [x] 将页面拆分为以下区域：
  - 日常维护：同步最新数据、查看历史版本等低风险动作。
  - 数据备份：下载、上传、历史版本、恢复。
  - 高级 JSON 编辑：完整 JSON 编辑和保存。
- [x] 高级 JSON 编辑默认折叠或增加明确的“高级模式”入口。
- [x] 高风险操作保留二次确认和数据变化摘要。

验收标准：

- [x] 用户进入页面后优先看到日常维护操作，而不是完整 JSON。
- [x] JSON 编辑入口有明确风险提示。
- [x] 备份恢复功能和日常维护功能视觉上分区清晰。

### P1：站点列表批量维护能力（已完成基础能力）

提升大量站点维护效率，减少逐个编辑。

当前状态：基础批量能力已完成，站点列表支持选中后启用/禁用、移动分类、设为私有和设为公开。后续可继续补充批量操作审计记录和结果明细。

功能要求：

- [x] 站点列表支持多选后的批量操作。
- [x] 支持批量启用/禁用。
- [x] 支持批量设置公开/私密。
- [x] 支持批量移动到指定一级分类或二级分类。

验收标准：

- [x] 用户能在站点列表中完成常见批量维护，不必进入 JSON 编辑。
- [x] 批量操作前展示影响数量。
- [x] 批量操作后提供结果反馈。

### P1：批量任务审计记录

为后台批量维护动作提供可追溯记录。

当前状态：未完成。站点列表批量操作目前只有即时 toast 反馈，没有持久化任务摘要和详情。

功能要求：

- [ ] 记录每次批量任务的摘要。
- [ ] 摘要包含：任务类型、触发人、开始时间、结束时间、处理数量、更新数量、失败数量。
- [ ] 保存最近若干次任务结果。
- [ ] 支持查看任务详情。

验收标准：

- [ ] 用户可以查看最近一次任务改动了什么。
- [ ] 失败列表不会因为页面刷新而丢失。

### P2：统一资源与图标缓存的产品表达（已完成）

降低用户对底层存储实现的理解成本。

当前状态：已完成。后台资源入口、资源页、网站图标下载页和资源错误信息已统一为面向管理员的产品表达；Blob、KV、Redis 等技术细节保留在系统状态和配置说明中。

功能要求：

- [x] 后台统一使用“图片资源”“自动缓存图标”等产品化命名。
- [x] 技术细节如 Blob、KV、Redis 只在配置说明或系统状态中展示。
- [x] 资源管理页明确区分“手动上传资源”和“自动缓存图标”。

验收标准：

- [x] 普通管理员不需要理解 Blob/KV 也能完成资源管理。
- [x] 开发部署者仍能在配置说明中看到底层存储状态。

### P2：系统状态卡片（已完成）

在后台增加系统状态入口，集中展示关键配置是否可用。

当前状态：已完成。后台新增“系统状态”入口，提供服务检查、后台能力检查、运行配置摘要和刷新状态能力。

功能要求：

- [x] 展示 Redis/KV 是否可用。
- [x] 展示 Vercel Blob 是否可用。
- [x] 展示当前图标和资源会写入的位置。
- [x] 展示历史版本能力是否可用。
- [x] 对缺失配置给出明确修复建议。

验收标准：

- [x] 用户可以在后台直接判断当前部署能力是否完整。
- [x] 配置缺失时，不需要通过失败操作反推问题原因。

## 推荐实施顺序

1. [x] 先完成数据管理页信息架构重构。
2. [x] 完成站点列表批量维护基础能力。
3. [ ] 补齐任务审计。（统一资源表达和系统状态卡片已完成）

## 设计原则

- 默认保护人工数据，不自动覆盖已有描述和图标。
- 批量写入后必须可追溯。
- 日常维护入口优先，JSON 编辑保留为高级能力。
- 技术配置对部署者透明，对普通管理员隐藏复杂度。
