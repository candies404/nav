# 需求文档

## 背景

NavSphere 已具备导航数据管理、站点管理、资源管理、历史版本、图标缓存和站点信息自动获取能力。深度使用后台后，当前主要问题不是能力缺失，而是部分维护流程仍偏工程化：用户能完成操作，但过程反馈、风险控制和批量维护效率还不够直观。

本文档记录后续面向管理员体验的改进需求。

## 用户痛点

2. 自动补全缺少预览与确认。
   - 站点描述来自网页 meta 或兜底逻辑，质量不稳定。用户需要在批量写入前预览结果，并能选择应用、跳过或重试。

3. 数据管理页职责过重。
   - 当前页面同时包含 JSON 编辑、保存、下载、上传、恢复历史和批量维护工具。普通管理员容易把它理解成“危险控制台”。

4. JSON 编辑入口风险感强。
   - 直接编辑完整导航 JSON 适合高级维护，但不适合日常运营。即使有校验和历史版本，也仍然容易误操作。

5. 站点列表批量操作不足。
   - 管理大量站点时，用户常需要批量补全描述、刷新图标、启用/禁用、移动分类、设置公开/私密等操作。

6. 批量任务结果不可追溯。
   - 任务执行完后，用户只能刷新数据查看最终状态，无法知道本次具体修改了哪些站点、哪些失败、失败原因是什么。

7. 资源库和自动缓存概念容易混淆。
   - 用户不关心 Blob、KV asset、favicon cache key 等实现细节，更希望统一理解为“图片资源”和“自动缓存图标”。

8. 系统配置状态不够直观。
   - Redis/KV、Vercel Blob 等配置缺失时，用户需要从错误信息判断能力是否可用。后台应直接展示当前存储能力和缺失项。

## 需求列表

### P0：自动补全结果预览与确认

避免批量补全描述或图标时直接写入不可控结果，先让用户看到候选内容。

当前状态：未完成。当前数据管理页和站点列表可触发补全，但不会在写入前展示候选描述、候选图标或逐项应用/跳过选择。

功能要求：

- [ ] 批量补全前生成候选结果列表。
- [ ] 展示每个站点的原描述、候选描述、原图标、候选图标和来源。
- [ ] 支持逐项应用、跳过或重新获取。
- [ ] 支持只应用描述、只应用图标或两者同时应用。

验收标准：

- [ ] 用户可以在写入前确认自动补全结果。
- [ ] 自动补全不会默认覆盖已有人工描述。
- [ ] 用户可以跳过质量不高的候选内容。

### P1：批量补全任务状态与结果面板

提升数据管理页中“补全图标与描述”后台任务的可见性。

当前状态：未完成。接口已能后台启动补全任务并在 toast 中提示队列数量，但页面没有任务状态面板，刷新页面后也看不到本次任务进度、失败列表和结果详情。

功能要求：

- [ ] 后台任务提供可查询的任务状态。
- [ ] 页面展示任务状态、进度、开始时间、结束时间和批次信息。
- [ ] 展示处理数量、更新图标数量、补全描述数量、失败数量。
- [ ] 展示失败站点列表和失败原因。
- [ ] 支持重试失败项。

验收标准：

- [ ] 用户可以在页面上看到当前或最近一次补全任务进度。
- [ ] 任务失败后可以定位具体站点和失败原因。
- [ ] 页面刷新后不会丢失最近一次任务结果。

### P1：数据管理页拆分为更清晰的维护区域（已完成基础能力）

降低数据管理页的认知负担，把高风险能力和日常维护能力区分开。

当前状态：基础拆分已完成，数据管理页按“日常维护 / 备份恢复 / 高级 JSON”分区；高级 JSON 作为独立高级入口，保存前展示数据变化摘要，高风险变更需要再次确认。

功能要求：

- [x] 将页面拆分为以下区域：
  - 批量维护：补全图标与描述、刷新站点元信息等。
  - 数据备份：下载、上传、历史版本、恢复。
  - 高级 JSON 编辑：完整 JSON 编辑和保存。
- [x] 高级 JSON 编辑默认折叠或增加明确的“高级模式”入口。
- [x] 高风险操作保留二次确认和数据变化摘要。

验收标准：

- [x] 用户进入页面后优先看到日常维护操作，而不是完整 JSON。
- [x] JSON 编辑入口有明确风险提示。
- [x] 备份恢复功能和批量维护功能视觉上分区清晰。

### P1：站点列表批量维护能力（已完成基础能力）

提升大量站点维护效率，减少逐个编辑。

当前状态：基础批量能力已完成，站点列表支持选中后补全描述、刷新图标、启用/禁用、移动分类、设为私有和设为公开。后续可继续补充批量操作审计记录和结果明细。

功能要求：

- [x] 站点列表支持多选后的批量操作。
- [x] 支持批量补全描述。
- [x] 支持批量刷新图标。
- [x] 支持批量启用/禁用。
- [x] 支持批量设置公开/私密。
- [x] 支持批量移动到指定一级分类或二级分类。

验收标准：

- [x] 用户能在站点列表中完成常见批量维护，不必进入 JSON 编辑。
- [x] 批量操作前展示影响数量。
- [x] 批量操作后提供结果反馈。

### P1：批量任务审计记录

为后台自动维护动作提供可追溯记录。

当前状态：未完成。站点列表批量操作和数据管理补全任务目前只有即时 toast 反馈，没有持久化任务摘要和详情。

功能要求：

- [ ] 记录每次批量任务的摘要。
- [ ] 摘要包含：任务类型、触发人、开始时间、结束时间、处理数量、更新数量、失败数量。
- [ ] 保存最近若干次任务结果。
- [ ] 支持查看任务详情。

验收标准：

- [ ] 用户可以查看最近一次任务改动了什么。
- [ ] 失败列表不会因为页面刷新而丢失。

### P2：统一资源与图标缓存的产品表达（已完成）

降低用户对底层存储实现的理解成本。

当前状态：已完成。后台资源入口、资源页、网站图标下载页和资源错误信息已统一为面向管理员的产品表达；Blob、KV、Redis 等技术细节保留在系统状态和配置说明中。

功能要求：

- [x] 后台统一使用“图片资源”“自动缓存图标”等产品化命名。
- [x] 技术细节如 Blob、KV、Redis 只在配置说明或系统状态中展示。
- [x] 资源管理页明确区分“手动上传资源”和“自动缓存图标”。

验收标准：

- [x] 普通管理员不需要理解 Blob/KV 也能完成资源管理。
- [x] 开发部署者仍能在配置说明中看到底层存储状态。

### P2：系统状态卡片（已完成）

在后台增加系统状态入口，集中展示关键配置是否可用。

当前状态：已完成。后台新增“系统状态”入口，提供服务检查、后台能力检查、运行配置摘要和刷新状态能力。

功能要求：

- [x] 展示 Redis/KV 是否可用。
- [x] 展示 Vercel Blob 是否可用。
- [x] 展示当前图标和资源会写入的位置。
- [x] 展示历史版本能力是否可用。
- [x] 对缺失配置给出明确修复建议。

验收标准：

- [x] 用户可以在后台直接判断当前部署能力是否完整。
- [x] 配置缺失时，不需要通过失败操作反推问题原因。

## 设计原则

- 默认保护人工数据，不自动覆盖已有描述。
- 批量写入前尽量可预览，写入后必须可追溯。
- 日常维护入口优先，JSON 编辑保留为高级能力。
- 技术配置对部署者透明，对普通管理员隐藏复杂度。
